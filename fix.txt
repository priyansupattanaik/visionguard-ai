## SECTION 1 — ROOT CAUSE FIXES IN pipeline.py

Fix all five root causes in this exact order. Do not skip any.

### FIX 1 — Query-time frame re-selection inside the matched window

PROBLEM:
The representative frame stored per window is chosen at scan time before any query exists.
When a top window is returned, the displayed frame is not the frame inside that window
that best matches the actual query. This is the primary reason wrong frames are shown.

WHAT TO ADD:
Add a private method to VisionGuardPipeline called _reselect_best_frame.

Signature:
    def _reselect_best_frame(self, video_path, start_sec, end_sec, query_vec, step_sec=0.1):

Logic:
1. Open the video with OpenCV VideoCapture.
2. Seek to the frame at start_sec using CAP_PROP_POS_MSEC.
3. Read frames at step_sec intervals until end_sec is exceeded.
4. For each frame read, call self.vlm.embed_frame(frame) to get its embedding.
5. Compute cosine similarity: float(np.dot(emb, query_vec))
6. Track the frame with the highest score, its timestamp (CAP_PROP_POS_MSEC / 1000.0),
   and save it as a temporary JPEG to the current run's frames/ folder with a name like
   resel_{start_sec:.2f}.jpg
7. Release the cap.
8. Return: (saved_frame_path, best_timestamp, best_score)

WHERE TO CALL IT:
In the search() method, after turbovec returns top window hits and before prepare_hits()
is called, loop over the top hits (maximum 4) and call _reselect_best_frame for each.
Replace the hit's representative_frame_path and timestamp with the re-selected values.

DO NOT call this on every window. Only the top hits that will be shown to the user.
DO NOT change any other part of the scan flow. scan-time representative frames stay as-is.

### FIX 2 — Cluster representative must be the highest-scoring frame in the cluster

PROBLEM:
When nearby high-scoring sampled frames are clustered into one clip candidate,
the representative frame chosen is not guaranteed to be the frame with the
highest individual similarity score. This causes the shown frame to be off.

WHERE TO FIX:
In pipeline.py, find the temporal clustering logic that groups nearby frames
into clip candidates and selects a representative for each cluster.

WHAT TO CHANGE:
Replace any current representative selection logic (midpoint, index 0, or any
other method) with:
    best_hit = max(cluster_members, key=lambda f: f['score'])
Use best_hit's frame_path as the cluster representative.
Use best_hit's timestamp as the cluster's moment.

Do not change how clusters are formed. Only change which frame becomes the representative.

### FIX 3 — Florence-2 phrase grounding at search display time for top-1 result

PROBLEM:
Florence-2 grounding currently runs only at export time.
The matched frame shown in the gallery after search has no bounding box drawn on it
unless the query maps to a YOLO detector class.
The user cannot confirm the frame is correct because there is no spatial indicator.

WHAT TO ADD:
In prepare_hits() or in search(), after the top-1 hit is resolved and its
representative frame path is known:

1. Call florence.ground_phrase(frame_path, original_query) which returns a list of boxes.
   This uses Florence-2-large with CAPTION_TO_PHRASE_GROUNDING task.
2. If boxes are returned:
   a. Draw the boxes onto the frame (green rectangles, line width 2).
   b. Save the result as a new JPEG: gallery_{hit_id}.jpg in the run's frames/ folder.
   c. Set hit['gallery_frame'] = that path.
3. If florence.ground_phrase returns empty or fails:
   a. Check if hit has stored YOLO det_boxes from scan time.
   b. If yes, draw those boxes and save as gallery_{hit_id}.jpg.
   c. If no boxes at all, use the raw representative frame as-is.

Do this ONLY for index 0 of the returned hits (the top result).
Hits 1–3 continue to show raw representative frames in the gallery (unchanged behavior).
This keeps latency acceptable while making the most important result visually confirmed.

The gallery display in app.py must use hit['gallery_frame'] if present,
otherwise fall back to hit['representative_frame_path'].

### FIX 4 — YOLO box fallback when Florence grounding returns empty

PROBLEM:
When Florence-2 grounding returns empty boxes or fails, the frame is shown with
no spatial indicator. For detector-supported queries (person, car, truck, bus,
motorcycle, bicycle, umbrella) the scan-time YOLO boxes are already stored but
never used at this stage.

WHAT TO ADD:
In the code path that handles Florence grounding output (both at search display time
per Fix 3 above, and at export time in segmenter.py):

After calling Florence grounding:
    boxes = florence.ground_phrase(frame_path, query)
    if not boxes:
        stored = hit.get('det_boxes', [])
        if stored:
            boxes = stored

det_boxes must already be stored per frame at scan time (they are, per documentation §5B.4).
Do not change the scan-time storage logic. Only add the fallback read at result display time.

### FIX 5 — Dense timestamp re-read is already covered by Fix 1

Fix 1 uses step_sec=0.1 inside _reselect_best_frame which gives 100ms frame precision
inside the matched window. No additional separate fix needed for timestamp precision.
Do not add any other sampling changes to the broad scan (sample_sec = 0.75 stays).

## SECTION 6 — UPDATE README.md

Rewrite README.md. It must contain exactly these sections in this order.
Keep it concise. No marketing language. No false capability claims.

### VisionGuard AI

A scan-first CCTV video search system.
Scan a video once. Search it many times using natural-language queries.
Returns matched frames with timestamps, bounding boxes on top results, and exportable clips.

### How It Works

1. Upload a video and click Scan.
2. The system samples frames, detects objects, and builds a searchable index.
3. Enter a natural-language query and click Find Matches.
4. The top matched frame is shown with a grounding box if the queried phrase can be located.
5. Export selected clips, CSV, JSON, or HTML reports.

### Model Stack

| Role | Model |
|---|---|
| Detection + Tracking | YOLO11m + BoT-SORT |
| Retrieval | SigLIP2 So400m/14 384 |
| Verification + Grounding | Florence-2-large |
| Segmentation | SAM2.1-hiera-small |
| Vector Index | turbovec IdMapIndex |

### What Works Well

- Natural-language search over a scanned video
- Repeated queries without rescanning
- Object-aware retrieval (person, car, truck, bus, motorcycle, bicycle, umbrella)
- Color-object queries for supported vehicle classes (yellow car, white bus)
- Grounding box on top-1 result at search time
- Exact frame re-selection within matched windows
- Segmented clip and region mask on export

### Known Limitations

- Temporal events (collisions, fights, falls) are not guaranteed to be correctly identified.
  The system retrieves visually similar frames, not verified event classifications.
- Queries shorter than 3 words or very abstract queries (unusual activity, suspicious thing)
  will return weaker results.
- Very long videos increase scan time linearly. This is expected behavior.
